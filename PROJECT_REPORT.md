# 基于 Agentic RAG 的电商客服智能问答系统项目报告

## 摘要

本项目实现了一个面向电商客服场景的智能问答系统。系统基于 Agentic RAG 技术路线，将传统“检索后生成”的 RAG 流程扩展为一个可决策、可调用工具、可多轮修正、可自我检查、可转人工兜底的 Agent 工作流。

项目的核心场景包括退换货政策、配送时效、发票服务、会员权益、常见故障判定和订单物流查询。系统不仅能够基于本地知识库回答常见客服问题，还能够识别订单类问题并调用模拟订单查询工具；当检索资料不足时，系统会对用户问题进行查询改写并再次检索；当用户要求人工客服或多轮尝试仍无法可靠回答时，系统会进入转人工流程。

本项目同时提供单页前端界面、FastAPI 后端服务、LangGraph Agent 状态图、本地 Chroma 向量知识库、SSE 流式输出、32 条人工标注测试集和 4 组消融实验脚本，可用于课程报告、答辩演示和 Agentic RAG 原型验证。

## 2. 项目目标和背景

### 2.1 项目背景

在电商客服场景中，用户问题通常具有以下特点：

- 表达方式口语化，例如“耳机坏了能退吗”“快递咋还没到”。
- 问题类型混杂，例如同一句话可能同时涉及产品故障和退换货政策。
- 部分问题需要查询业务系统，例如订单状态、物流进度。
- 客服回答需要严格遵守业务政策，不能随意承诺或编造。
- 用户可能随时要求人工客服介入。

传统 FAQ 或关键词匹配机器人难以处理复杂问法。普通 RAG 虽然能引入知识库，但通常采用固定流程：

```text
用户问题 -> 检索知识库 -> 大模型生成答案
```

这种流程在以下场景中会出现明显不足：

- 订单问题不应该查知识库，而应该查订单系统。
- 第一次检索失败后，系统不会主动换一种方式重新检索。
- 检索到的文档可能不相关，但生成模型仍可能参考它们。
- 最终答案可能超出资料依据。
- 系统内部决策不可见，不利于教学、调试和评估。

因此，本项目采用 Agentic RAG 架构，将客服 AI 从“只会检索回答”的问答系统，扩展为“会判断、会行动、会修正、会自检、会兜底”的智能客服 Agent。

### 2.2 项目目标

本项目的目标分为业务目标、技术目标和教学目标。

业务目标：

- 构建一个可运行的电商客服问答系统原型。
- 支持退换货、配送、发票、会员、故障判定等知识库问答。
- 支持订单号相关问题的工具调用。
- 支持闲聊、业务范围外问题和转人工请求。
- 在资料不足时避免强行回答，给出人工兜底。

技术目标：

- 使用 LangGraph 显式编排 Agent 状态图。
- 使用 LangChain + ChromaDB 构建本地 RAG 知识库。
- 通过 OpenAI 兼容协议接入智谱 GLM 等大模型服务。
- 使用 SSE 实现后端流式输出和前端实时展示。
- 通过消融实验验证不同 Agentic 组件的作用。

教学目标：

- 让非专业读者理解 Agentic RAG 的应用价值。
- 让专业读者看到完整的工程结构和评测闭环。
- 为课程报告和答辩提供清晰、具体、可复现的项目材料。

## 3. 项目要解决的问题

### 3.1 只靠知识库检索无法处理所有客服问题

用户问“订单 A1001 到哪了”时，正确做法是查询订单或物流系统，而不是从退换货政策、配送时效等文档中找答案。朴素 RAG 如果不具备路由能力，会把订单类问题错误地当作知识库问答处理。

本项目通过 `router` 节点将用户请求分为：

- `kb`：知识库问答。
- `tool`：工具调用，例如订单查询。
- `chitchat`：闲聊或业务范围外问题。
- `handoff`：用户要求转人工或投诉升级。

### 3.2 一次检索无法覆盖复杂问题

例如“我买的耳机突然没声音，能退吗？”这个问题至少涉及两个知识点：

- 耳机没声音是否属于质量问题。
- 质量问题是否支持退换货。

如果系统只检索一次，可能只命中故障文档或只命中退换货文档，导致答案不完整。本项目通过 `grade_docs` 和 `rewrite_query` 形成多轮检索机制，提高复杂问题的资料覆盖率。

### 3.3 大模型可能产生无依据回答

客服系统不能凭常识承诺退款、赔偿、时效或售后政策。项目通过生成约束和 `reflect` 节点检查答案是否忠于资料，降低幻觉风险。

### 3.4 系统过程不可见会影响信任和评估

如果只显示最终答案，用户和评审很难知道系统为什么这么回答。本项目在前端右侧展示 Agent 的实时执行过程，包括路由、检索、打分、改写、工具调用、生成和反思步骤。

## 4. 面向非专业人士的项目解释

可以把本项目理解成一个“带工作流程的客服助理”。

普通机器人像一本会搜索的说明书。用户问问题，它就在说明书里找相似内容，然后组织语言回答。

本项目中的客服 AI 更像一个经过培训的客服人员：

1. 先听懂用户大概想做什么。
2. 如果是政策问题，就去查客服知识库。
3. 如果是订单问题，就去查订单工具。
4. 如果用户说“我要转人工”，就直接转人工。
5. 如果第一次资料不够，就换一种更标准的说法再查一次。
6. 回答时尽量只依据已有政策和工具结果。
7. 无法可靠回答时，不硬编答案，而是建议人工处理。

举例：

| 用户问题 | 系统应做的事 |
| --- | --- |
| “退款多久能到账？” | 查询退换货政策文档 |
| “订单 A1001 到哪了？” | 调用订单查询工具 |
| “耳机没声音能退吗？” | 同时查故障判定和退换货政策 |
| “你们 CEO 是谁？” | 礼貌说明当前系统不处理公司信息 |
| “我要转人工” | 进入人工客服转接流程 |

本项目的一个重要特点是：它不只给出答案，还会把“它刚才做了什么”展示出来，因此更适合演示、教学和答辩。

## 5. RAG 与 Agentic RAG 概念说明

### 5.1 RAG 是什么

RAG 是 Retrieval-Augmented Generation，即“检索增强生成”。

它的基本流程是：

```text
用户问题 -> 从知识库检索相关资料 -> 将资料交给大模型 -> 生成回答
```

RAG 的价值在于：

- 大模型不必完全依赖自身记忆。
- 回答可以基于企业自己的文档。
- 业务文档更新后，可以通过重建索引更新知识。
- 回答更容易追溯依据。

### 5.2 普通 RAG 的局限

普通 RAG 通常流程固定，不会根据问题类型动态改变行为。它无法很好地区分：

- 需要查知识库的问题。
- 需要查业务系统的问题。
- 需要转人工的问题。
- 不属于业务范围的问题。

另外，普通 RAG 对“第一次检索失败”的处理能力较弱，也缺少对最终答案的自检。

### 5.3 Agentic RAG 是什么

Agentic RAG 可以理解为“带智能决策流程的 RAG”。

它不是简单执行“检索 -> 生成”，而是让系统具备类似 Agent 的能力：

- 先判断任务类型。
- 决定使用知识库还是工具。
- 判断检索结果是否足够。
- 必要时改写查询并多轮检索。
- 对最终答案进行忠实度检查。
- 失败时进入人工兜底。

本项目的 Agentic RAG 流程如下：

```text
用户问题
  -> router 判断意图
  -> retrieve 检索知识库 或 tool_call 调用工具
  -> grade_docs 判断资料是否相关
  -> rewrite_query 必要时改写查询
  -> generate 生成答案
  -> reflect 检查答案忠实度
  -> done 或 human_handoff
```

## 7. Agent 工作流和状态流程

### 7.1 Agent 状态图

项目使用 LangGraph 将整个客服 Agent 编排为状态图：

```text
START
  │
  ▼
router
  ├── kb ───────────────► retrieve ─► grade_docs
  │                                      │
  │                         ┌────────────┴────────────┐
  │                         │                         │
  │                      资料足够                   资料不足
  │                         │                         │
  │                         ▼                         ▼
  │                      generate              rewrite_query
  │                         │                         │
  │                         ▼                         │
  │                      reflect ◄────────────────────┘
  │                         │
  │                         ▼
  │                        END
  │
  ├── tool ─────────────► tool_call ─► generate ─► END
  │
  ├── chitchat ─────────► generate ─► END
  │
  └── handoff ──────────► human_handoff ─► END
```

### 7.2 状态对象

Agent 的状态定义在 `app/agent/state.py` 中。核心字段包括：

| 字段 | 作用 |
| --- | --- |
| `question` | 用户原始问题 |
| `route` | 路由结果：`kb`、`tool`、`chitchat`、`handoff` |
| `tool_name` / `tool_args` | 工具名称和参数 |
| `rewritten` | 改写后的查询 |
| `docs` | 检索到的文档片段 |
| `grades` | 每个文档片段的相关性判断 |
| `retries` | 查询改写和重试次数 |
| `answer` | 最终回答 |
| `citations` | 引用来源 |
| `tool_result` | 工具调用结果 |
| `grounded` | 反思节点判断答案是否忠实 |
| `steps` | 前端时间线和评测记录使用的步骤信息 |
| `end_reason` | 结束原因，例如 `answered` 或 `handoff` |

### 7.3 四种 Agent 变体

项目在 `app/agent/graph.py` 中定义了四种实验变体：

| 变体 | 说明 |
| --- | --- |
| `naive` | E0 朴素 RAG，无路由、无工具、无打分、无改写、无反思、无重排 |
| `e1` | 加入路由、工具调用和检索重排 |
| `e2` | 在 E1 基础上加入文档打分和查询改写 |
| `full` | 完整 Agentic RAG，包含反思节点 |

这种设计可以在前端直接切换，也可以通过评测脚本批量对比。

## 8. 技术方案与技术选型

| 技术 | 项目用途 | 选择原因 |
| --- | --- | --- |
| Python | 后端和 AI 工程主语言 | AI 生态成熟，适合 LangChain / LangGraph |
| FastAPI | Web 服务框架 | 轻量、异步友好、易于提供 API |
| LangGraph | Agent 状态图编排 | 能显式控制节点和条件边，便于观察和评测 |
| LangChain | 文档、模型、Embedding、向量库集成 | RAG 生态完善 |
| ChromaDB | 本地向量数据库 | 部署简单，适合课程 Demo 和本地实验 |
| langchain_openai | OpenAI 兼容模型接入 | 可接入 OpenAI 官方、智谱 GLM 等兼容服务 |
| sse-starlette | SSE 流式响应 | 前端可实时接收步骤和 token |
| RAGAS | 生成质量评估 | 可评估忠实度、相关性和上下文精度 |

当前项目默认使用智谱 GLM 的 OpenAI 兼容接口：

```text
LLM_MODEL=glm-4-flash
EMBED_MODEL=embedding-3
```

注意：报告不包含真实 API Key。实际运行时应在 `.env` 中配置密钥，提交项目时应删除真实 `.env`。

## 9. 后端模块设计

### 9.1 FastAPI 入口

文件：

```text
app/main.py
```

职责：

- 创建 FastAPI 应用。
- 注册 CORS 中间件。
- 注册 `/api` 路由。
- 提供 `/health` 健康检查。
- 提供 `/` 前端页面。
- 启动时检查 `OPENAI_API_KEY` 是否为空，避免第一次请求时才报错。

### 9.2 配置模块

文件：

```text
app/config.py
```

通过 `pydantic-settings` 从 `.env` 读取配置：

| 配置项 | 说明 |
| --- | --- |
| `OPENAI_API_KEY` | 模型服务密钥，占位配置，不应提交真实值 |
| `OPENAI_BASE_URL` | OpenAI 兼容接口地址 |
| `LLM_MODEL` | 对话模型名称 |
| `EMBED_MODEL` | Embedding 模型名称 |
| `TOP_K` | 检索返回文档数 |
| `MAX_RETRIES` | 最大改写重试次数 |
| `CHROMA_DIR` | Chroma 本地存储目录 |
| `DOCS_DIR` | 知识库文档目录 |
| `HOST` / `PORT` | 服务监听地址和端口 |

当前版本的 `get_settings()` 不做缓存，这样开发阶段修改 `.env` 后更容易生效。

### 9.3 聊天 API

文件：

```text
app/api/chat.py
```

核心接口：

```text
POST /api/chat
```

请求字段：

| 字段 | 说明 |
| --- | --- |
| `question` | 用户问题 |
| `variant` | Agent 变体，支持 `naive`、`e1`、`e2`、`full` |

接口内部通过 `graph.astream_events()` 监听 LangGraph 执行事件，并转成 SSE 返回给前端。

另一个接口：

```text
POST /api/ingest?rebuild=true
```

用于前端触发知识库重建。

## 12. 文档加载、切分、Embedding 和向量库构建

### 12.1 文档加载

文件：

```text
app/ingestion/loader.py
```

系统会读取 `DOCS_DIR` 下的 `.md` 和 `.txt` 文件，并为每个文档生成 LangChain `Document`。

保留的元数据：

- `source`：文件名，用于前端引用。
- `path`：相对路径。

### 12.2 文档切分

文件：

```text
app/ingestion/chunker.py
```

切分采用两阶段策略：

1. Markdown 标题切分：根据 `#`、`##`、`###` 保留章节结构。
2. 递归字符切分：将长文本切成适合 Embedding 的小片段。

默认参数：

```text
chunk_size = 400
chunk_overlap = 60
```

分隔符优先级包括：

```text
空行、换行、句号、分号、逗号、空格
```

这样做可以兼顾中文文本结构和向量检索效果。

### 12.3 Embedding 模型

文件：

```text
app/ingestion/embedder.py
```

当前通过 `OpenAIEmbeddings` 接入 OpenAI 兼容 Embedding 服务。默认配置为智谱：

```text
EMBED_MODEL=embedding-3
```

项目将 Embedding 工厂集中在一个文件中，便于未来切换为本地模型或其他服务商。

### 12.4 Chroma 向量库

文件：

```text
app/ingestion/vectorstore.py
```

构建流程：

```text
加载文档 -> 切分文档 -> 计算 Embedding -> 写入 ChromaDB
```

默认持久化目录：

```text
.chroma/
```

构建命令：

```bash
python -m scripts.ingest
```

当前脚本默认清空并重建索引，避免重复运行时向 Chroma 追加重复 chunk。

## 13. 检索与重排方案

### 13.1 基础检索

文件：

```text
app/retrieval/retriever.py
```

系统使用 Chroma 的 `similarity_search_with_score()` 进行向量相似度检索，返回：

```text
(Document, distance)
```

其中 distance 越小表示越相似。

### 13.2 检索管线

文件：

```text
app/retrieval/pipeline.py
```

E1、E2、E3 使用完整检索管线：

```text
query -> retrieve_with_scores -> rerank -> top-k documents
```

为了给重排更多候选，系统会先多取候选文档：

```text
over_fetch = top_k * 3
```

### 13.3 重排策略

文件：

```text
app/retrieval/reranker.py
```

当前重排不是深度学习 reranker，而是一个轻量、可解释的策略：

1. 去重：根据 `source + section + 文本前 100 字` 去除重复片段。
2. 来源多样性惩罚：同一个来源多次出现时增加分数惩罚，避免 top-k 被同一文档占满。

这种设计有利于多跳问题，因为多跳问题往往需要多个文档共同支持答案。

### 13.4 E0 的朴素检索

E0 使用 `retrieve_node_plain`，即纯向量 top-k，不做重排。这使 E0 能作为真正的朴素 RAG 基线，便于比较 Agentic 组件带来的提升。

## 14. Agent 各节点设计

### 14.1 router：意图路由

职责：

判断用户问题属于哪条路径。

路由类别：

| route | 场景 | 示例 |
| --- | --- | --- |
| `kb` | 知识库问题 | “退款多久能到账？” |
| `tool` | 订单或物流查询 | “订单 A1001 到哪了？” |
| `chitchat` | 闲聊或业务范围外 | “你好”“你们 CEO 是谁？” |
| `handoff` | 主动要求人工 | “我要转人工”“我要投诉” |

实现特点：

- 模型被要求输出 JSON。
- Python 端使用防御性解析。
- 不强依赖 function calling 或 structured output，增强对 GLM 等兼容服务商的适配性。
- 如果模型未输出合法 route，默认回退到 `kb`。

### 14.2 retrieve：知识库检索

职责：

根据原始问题或改写后的查询，从 ChromaDB 检索相关文档。

E1、E2、E3 使用带重排的 `retrieve_node`。E0 使用不带重排的 `retrieve_node_plain`。

### 14.3 grade_docs：文档相关性判断

职责：

对每个检索片段判断是否能回答用户问题。

输出为布尔列表：

```text
[true, false, true, ...]
```

如果没有保留任何相关文档，且仍有重试次数，则进入查询改写。

### 14.4 rewrite_query：查询改写

职责：

将用户原始问题改写为更适合向量检索的查询。

改写策略包括：

- 将口语化表达转换为业务术语。
- 展开同义词。
- 保留核心检索关键词。
- 面向中文电商客服领域优化表达。

示例：

```text
原问题：我买的耳机突然没声音，能退吗？
可能改写：耳机无声音 故障判定 质量问题 退换货政策
```

### 14.5 tool_call：工具调用

文件：

```text
app/agent/tools.py
```

当前工具：

```text
mock_order_lookup
```

支持订单：

- `A1001`
- `A1002`
- `A1003`
- `A1004`

工具返回：

- 订单状态。
- 物流公司。
- 运单号。
- 预计送达时间或签收时间。
- 商品列表。

这展示了 Agentic RAG 的关键能力：当问题需要业务系统数据时，Agent 不应强行检索知识库，而应调用工具。

### 14.6 generate：答案生成

职责：

根据文档资料和工具结果生成最终客服回答。

生成策略：

- `chitchat` 路径使用自然口吻回答，不强制引用知识库。
- `kb` 和 `tool` 路径使用严格 grounding prompt。
- 如果有工具结果，将工具结果自然融入回答。
- 如果资料不足，建议转人工。

该节点启用 streaming，并通过 `tags=["generate"]` 让 API 层只捕获生成节点的 token 流。

### 14.7 reflect：答案忠实度检查

职责：

判断答案是否完全基于提供的资料，没有编造。

如果答案不忠实，且尚未超过重试次数，可以回到查询改写流程。

### 14.8 human_handoff：转人工

触发方式：

1. 用户主动要求转人工。
2. 多轮检索和改写后仍没有足够资料。

系统会输出人工客服转接话术，并提供 400 电话和响应承诺。

## 15. SSE 流式输出机制

### 15.1 为什么使用 SSE

客服对话需要实时反馈。如果后端等完整 Agent 流程执行完才返回，用户会感觉系统卡顿。

SSE 可以让后端持续推送事件：

- Agent 当前执行到哪一步。
- 大模型正在生成的 token。
- 最终答案和引用。

### 15.2 SSE 事件类型

后端在 `app/api/chat.py` 中输出以下事件：

| 事件 | 含义 |
| --- | --- |
| `step` | 某个 Agent 节点执行结束，包含节点输入输出摘要 |
| `token` | `generate` 节点产生的流式文本 |
| `done` | 整轮对话完成，返回最终答案、引用和结束原因 |
| `error` | 执行过程中出现异常 |

### 15.3 前端处理方式

前端使用 `fetch()` 读取响应流，并手动解析 SSE block。收到事件后：

- `step` 更新右侧时间线。
- `token` 追加到 AI 回答气泡中。
- `done` 渲染引用和结束状态。
- `error` 显示错误信息。

## 19. 指标说明

### 19.1 路由准确率

衡量系统是否选择了正确路径：

```text
route_accuracy = 正确路由数量 / 总样例数量
```

例如订单问题应走 `tool`，转人工问题应走 `handoff`。

### 19.2 工具 F1

衡量系统是否正确调用工具。

- Precision：调用工具的请求中，有多少确实应该调用。
- Recall：应该调用工具的请求中，有多少真的调用了。
- F1：Precision 和 Recall 的综合指标。

### 19.3 Hit@K

衡量期望来源文档是否出现在前 K 个检索结果中。

例如问题期望命中 `退换货政策.md`，如果该文档出现在 top-3，则 Hit@3 为命中。

### 19.4 MRR

MRR 是 Mean Reciprocal Rank，衡量正确文档排得有多靠前。

如果正确文档排第 1，得分为 1；排第 2，得分为 1/2；排第 3，得分为 1/3。

### 19.5 RAGAS 指标

项目可选使用 RAGAS 评估生成质量：

| 指标 | 含义 |
| --- | --- |
| Faithfulness | 答案是否忠实于检索上下文 |
| Answer Relevancy | 答案是否切题 |
| Context Precision | 检索上下文是否相关 |

### 19.6 延迟

记录端到端响应时间，包括：

- p50：一半请求低于该耗时。
- p95：95% 请求低于该耗时。

Agentic RAG 通常会比 E0 慢，因为它包含更多 LLM 调用和控制步骤。

## 22. 项目亮点

### 22.1 Agent 流程显式可控

项目没有使用完全黑盒的 Agent，而是通过 LangGraph 明确控制每个节点和条件边，适合客服这种要求稳定性的场景。

### 22.2 同时面向演示和评测

前端可以展示 Agent 思考过程，评测脚本可以记录每条样例的路由、检索、工具调用和延迟，形成演示与实验闭环。

### 22.3 兼容 OpenAI 协议生态

项目保留 `OPENAI_API_KEY`、`OPENAI_BASE_URL` 等通用变量，可在 OpenAI 官方、智谱 GLM 或其他兼容服务之间切换。

### 22.4 结构化输出兼容性更强

项目没有强依赖模型的 function calling，而是采用 prompt 约束输出 + Python 防御性解析，增强了对不同模型服务商的兼容性。

### 22.5 知识库构建与检索逻辑分层清晰

文档加载、切分、Embedding、向量库、检索、重排、查询改写分别放在独立模块中，便于替换和扩展。

## 23. 局限性

### 23.1 知识库规模较小

当前知识库只有 5 份示例文档，适合课程 Demo，但不足以代表真实企业知识库。

### 23.2 订单工具是模拟工具

当前 `mock_order_lookup` 使用硬编码订单数据，不连接真实订单系统。实际生产环境需要接入真实后端 API，并考虑权限、认证和审计。

### 23.3 检索重排较轻量

当前 reranker 是基于去重和来源多样性的启发式策略，没有使用 cross-encoder 等更强语义重排模型。

### 23.4 测试集规模有限

32 条测试样例足够展示课程项目，但不足以证明生产级鲁棒性。

### 23.5 模型输出仍有不确定性

即使有 JSON 解析和反思节点，大模型仍可能出现：

- 路由错误。
- yes/no 判断不稳定。
- 查询改写方向不理想。
- 生成答案过度保守或不够完整。

### 23.6 Agentic 流程增加延迟

E3 的步骤最多，LLM 调用次数也最多，因此延迟明显高于 E0。实际系统需要根据业务场景平衡准确率和响应速度。

## 25. 答辩讲解建议

### 25.1 推荐讲解顺序

1. 说明普通客服机器人和朴素 RAG 的问题。
2. 引出 Agentic RAG 的核心思想。
3. 介绍本项目的客服场景和知识库。
4. 展示系统架构图和 Agent 状态图。
5. 现场演示四类问题：政策、订单、多跳、转人工。
6. 解释 E0 到 E3 的消融实验设计。
7. 展示实验结果和典型案例。
8. 总结亮点、局限和未来工作。

### 25.2 面向非专业评委的讲法

可以这样讲：

> 这个项目不是简单的聊天机器人，而是一个有客服工作流程的 AI。它会先判断用户的问题类型，再决定查知识库、查订单工具、重新组织问题再查，或者转人工。右侧面板会把每一步展示出来，所以我们能看到它为什么这么回答。

### 25.3 面向专业评委的讲法

可以这样讲：

> 本项目使用 LangGraph 将 Agentic RAG 拆为 router、retrieve、grade、rewrite、tool、generate、reflect、handoff 等节点。知识库部分通过 Markdown 加载、标题切分、递归字符切分、Embedding 和 ChromaDB 持久化构建。检索阶段采用 over-fetch、去重和来源多样性重排。实验部分设计 E0 到 E3 四组消融，用路由准确率、工具 F1、Hit@K、MRR、RAGAS 和延迟评估各模块贡献。

### 25.4 演示时的注意事项

- 上传项目前删除真实 `.env`。
- 演示前确认 `.chroma/` 已构建。
- 先跑 `/health` 或打开首页确认服务正常。
- 建议先用 E3 展示完整流程，再切 E0 做对比。
- 订单演示推荐使用 `A1001`、`A1002`、`A1003` 或 `A1004`。

## 26. 项目结论

本项目实现了一个完整的 Agentic RAG 电商客服智能问答系统。它在普通 RAG 的基础上加入路由、工具调用、文档打分、查询改写、自我反思和人工兜底，使系统从“只会检索回答”升级为“能根据问题类型选择行动路径”的客服 Agent。

从工程实现看，项目结构清晰，前后端可运行，Agent 状态可观察，知识库构建与检索模块解耦，模型服务可通过 OpenAI 兼容协议替换。从实验设计看，项目提供了 32 条人工标注测试集和 4 组消融实验，能够较清楚地展示各 Agentic 组件的作用。

当前实验结果显示，加入路由和工具调用后，系统在订单类问题上的表现显著提升；加入文档打分和查询改写后，检索覆盖有所改善；加入反思后，系统具备了进一步控制答案忠实度的能力。与此同时，完整 Agentic RAG 的延迟高于朴素 RAG，说明实际落地时需要在响应速度、准确性和可信度之间做权衡。

总体而言，本项目适合作为 Agentic RAG 技术路线的课程项目、答辩演示和客服智能化原型，也为后续接入真实业务系统、扩大知识库和完善评测体系提供了清晰基础。
